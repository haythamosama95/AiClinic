## AI Architecture

### Isolation Principle

The AI service is a fully isolated layer:

```
┌──────────────┐      HTTP       ┌──────────────┐
│   Flutter     │◄──────────────►│  AI Service   │
│   Frontend    │  JSON request  │  (Ollama +    │
│               │  JSON response │   HTTP API)   │
└──────┬───────┘                 └──────────────┘
       │                                │
       │ Supabase SDK                   │ NEVER connects
       ▼                                │ to Supabase
┌──────────────┐                        │
│  Supabase    │◄───── no connection ───┘
│  Backend     │
└──────────────┘
```

- The AI service has NO database credentials.
- The AI service has NO Supabase URL or keys.
- The AI service receives prompts and optional context from the Flutter app, and returns structured JSON commands.
- The backend sees all requests as normal Supabase SDK calls from Flutter. It has zero awareness of AI.

### AI Service Architecture

#### Runtime

- **Ollama** as the inference runtime (supports llama.cpp, runs CPU-only, supports GGUF model format).
- A thin **HTTP wrapper service** sits in front of Ollama to provide:
  - Domain-specific system prompts per agent type
  - Request routing to the correct agent
  - Response parsing and structured output extraction
  - Request/response logging (local file, no DB)

#### Deployment

- Runs on the receptionist PC as a system service or Docker container.
- Listens on a configurable LAN port (default: `8090`).
- Accessible from any device on the clinic LAN.

#### Resource Constraints

| Constraint            | Specification                                                                         |
| --------------------- | ------------------------------------------------------------------------------------- |
| Max RAM for AI models | 3 GB (on 8GB systems), 4 GB (on 16GB systems)                                         |
| GPU                   | Not required. CPU inference only.                                                     |
| Context window        | Limited. No long-term memory, no cross-session context.                               |
| Model size            | 1B-7B parameter quantized models (Q4/Q5 GGUF)                                         |
| Concurrent requests   | Sequential processing. One request at a time per agent. Queuing for multiple clients. |

#### Recommended Models (CPU, 8GB RAM)

| Use Case                           | Model Class                   | Approx RAM  |
| ---------------------------------- | ----------------------------- | ----------- |
| Intent parsing, command generation | Phi-3 mini (3.8B, Q4)         | ~2.5 GB     |
| SOAP summarization                 | Phi-3 mini or Mistral 7B (Q4) | ~2.5 - 4 GB |
| Analytics query generation         | Phi-3 mini (3.8B, Q4)         | ~2.5 GB     |

On 8GB RAM systems, a single model is loaded. Agent switching reuses the same model with different system prompts.

### Internal Agent Model

Users interact with a unified chat interface. Internally, the AI service routes requests to specialized agents based on intent classification.

| Agent                | Responsibility                                  | Input                                        | Output                                                                                                     |
| -------------------- | ----------------------------------------------- | -------------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| **Scheduling Agent** | Appointment booking, rescheduling, cancellation | Natural language + current date/time context | `create_appointment`, `reschedule_appointment`, `cancel_appointment`, `update_appointment_status` commands |
| **Billing Agent**    | Invoice generation, payment recording           | Natural language + visit context             | `create_invoice`, `apply_payment` commands                                                                 |
| **SOAP Summarizer**  | Draft SOAP notes from free-text                 | Doctor's free-text description               | Structured SOAP JSON (subjective, objective, assessment, plan)                                             |
| **Analytics Agent**  | Answer operational questions                    | Natural language question                    | Pre-defined analytics query identifier + parameters                                                        |
| **Shift Agent**      | Shift scheduling assistance                     | Natural language + staff/branch context      | `create_shift`, `assign_staff` commands                                                                    |

#### Agent Routing

The AI service performs a two-step process:

1. **Intent Classification**: determine which agent should handle the request. This can be a simple keyword/pattern-based router or a lightweight model call.
2. **Agent Execution**: the selected agent's system prompt is loaded, context is assembled, and the model generates a structured response.

### Structured Command Protocol

All AI outputs follow a strict JSON schema. The AI service never returns free-text for actionable operations.

```json
{
  "command_type": "create_appointment",
  "confidence": 0.92,
  "params": {
    "patient_name": "Ahmed Hassan",
    "doctor_name": "Dr. Ali",
    "date": "2026-05-14",
    "time": "17:00",
    "type": "planned"
  },
  "display_summary": "Book Ahmed Hassan with Dr. Ali tomorrow at 5:00 PM",
  "requires_resolution": {
    "patient_id": "lookup_required",
    "doctor_id": "lookup_required"
  }
}
```

The Flutter app:
1. Receives the command.
2. Resolves any `lookup_required` fields (e.g., searches for patient by name, presents disambiguation if multiple matches).
3. Displays a confirmation card showing what will happen.
4. On user approval, executes the corresponding `supabase.rpc()` call.
5. On user rejection, discards the command.

### Context Strategy

AI agents are stateless and lightweight. Context is provided per-request by the Flutter app:

| Context Type                        | Source                                  | Example                              |
| ----------------------------------- | --------------------------------------- | ------------------------------------ |
| Current branch                      | Flutter app state                       | `branch_id`, `branch_name`           |
| Current date/time                   | System clock                            | `2026-05-14T17:00:00+02:00`          |
| Active patient (if in patient view) | Flutter UI state                        | `patient_name`, `patient_id`         |
| Doctor list (for the branch)        | Pre-fetched from Supabase, passed to AI | `[{"name": "Dr. Ali", "id": "..."}]` |
| Recent appointments (if relevant)   | Pre-fetched from Supabase, passed to AI | Limited list of today's appointments |

Context is assembled by the Flutter app before sending the prompt. The AI service never fetches context on its own.
